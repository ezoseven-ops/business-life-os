## 1. IMPLEMENTATION PHASES

Six phases, ordered from safest (zero user impact) to most visible (new UI). Each phase is independently deployable. If any phase fails, the system continues working exactly as it does today.

```
PHASE 0: Security Hardening           (0 risk, 0 UI change)
PHASE 1: Schema Migration             (0 risk, 0 UI change)
PHASE 2: Access Control Enforcement   (low risk, 0 UI change for OWNER/TEAM)
PHASE 3: Invitation System            (no risk to existing, new pages only)
PHASE 4: Client Portal                (no risk to existing, new layout only)
PHASE 5: Owner Dashboard Upgrade      (replaces home content, OWNER/TEAM only)
```

### Why This Order

Phase 0 fixes existing security holes that should be fixed regardless of RBAC. Phase 1 adds database tables and columns but changes no code behavior. Phase 2 wires up permission checks but existing OWNER/TEAM users pass all checks — they see no difference. Phase 3 adds new pages that don't exist yet. Phase 4 adds a new layout group. Phase 5 is the only phase that changes existing UI.

No phase touches existing working UI until Phase 5 — and even then, the change is additive (new dashboard sections), not destructive (nothing is removed).

### PHASE 2: Access Control Enforcement

**Goal:** Add role-checking utilities and wire them into every server action. OWNER and TEAM users see no difference — they pass all checks. This phase prepares the system for CLIENT users who will arrive in Phase 4.

#### 2A. New Utility Functions

| File | Change |
|------|--------|
| `src/lib/action-utils.ts` | Add `requireRole(minimumRole)` — checks role hierarchy OWNER > TEAM > CLIENT. Add `requireProjectAccess(projectId)` — for CLIENT checks ProjectMember; for OWNER/TEAM passes through. Add `requireTaskAccess(taskId)` — combines workspace check + role check + project membership for CLIENT. |

#### 2B. Server Action Permission Gates

Every existing action file gets a tighter guard. The change pattern is the same everywhere: replace `requireWorkspace()` with the appropriate new guard.

| File | Current Guard | New Guard |
|------|--------------|-----------|
| `src/modules/tasks/task.actions.ts` — `createTaskAction` | `requireWorkspace()` | `requireRole('TEAM')` — Clients cannot create tasks |
| `src/modules/tasks/task.actions.ts` — `updateTaskAction` | `requireWorkspace()` | `requireWorkspace()` + inline check: if CLIENT, only allow status change on assigned tasks |
| `src/modules/tasks/task.actions.ts` — `deleteTaskAction` | `requireWorkspace()` | `requireRole('TEAM')` + check: TEAM can only delete own; OWNER can delete any |
| `src/modules/notes/note.actions.ts` — all actions | `requireWorkspace()` | `requireRole('TEAM')` — Clients cannot create/edit/delete notes |
| `src/modules/events/event.actions.ts` — all actions | `requireWorkspace()` | `requireRole('TEAM')` — Clients cannot create/edit/delete events |
| `src/modules/people/people.actions.ts` — all actions | `requireWorkspace()` | `requireRole('TEAM')` — Clients cannot access CRM |
| `src/modules/communications/comms.actions.ts` — all actions | `requireWorkspace()` | `requireRole('TEAM')` — Clients cannot access inbox |
| `src/modules/comments/comment.actions.ts` — `addCommentAction` | `requireWorkspace()` | `requireWorkspace()` + inline check: if CLIENT, verify task access and force `internal: false` |
| `src/modules/projects/project.actions.ts` — create/update/delete | `requireWorkspace()` | `requireRole('TEAM')` |
| `src/modules/workspace/workspace.actions.ts` — `updateWorkspaceAction` | `requireOwner()` | No change (already correct) |

## 5. RISK MAP

### PHASE 0: Security Hardening

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| `getById` with workspace filter returns null for legitimate requests if workspaceId mismatch | Low | Medium — user sees "not found" | Verify the workspace ID is correctly passed from session in every page. Unit test: create task in workspace A, verify workspace B user gets null. |
| Turbopack cache after file changes | Medium | Low — stale page | Hard refresh or server restart. Known issue. |

### PHASE 1: Schema Migration

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| MEMBER→TEAM rename breaks code references | Medium | High — compile errors | Before migration: global search for `MEMBER` in all .ts/.tsx files. Replace with `TEAM`. Run `tsc --noEmit` to verify. |
| Migration fails on existing data | Low | High — DB in bad state | Run migration on a copy of the DB first. Backup before migrating production. |
| Prisma generate + Turbopack stale cache | High | Medium — dev server freeze | Documented in ARCHITECTURE-RBAC.md. Script: `rm -rf .next && npm run dev` after `prisma generate`. |
| Backfill script misses edge cases (user with no workspace, project with no workspace) | Low | Medium — some users lack ProjectMember records | Add WHERE clause: only users with workspaceId, only projects in same workspace. Log skipped records. |

### PHASE 2: Access Control Enforcement

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Existing OWNER/TEAM users blocked by overly strict guard | Low | High — broken actions | `requireRole('TEAM')` passes for both TEAM and OWNER (hierarchy check). Unit test: verify OWNER passes all guards, TEAM passes all except `requireOwner()`. |
| CLIENT query path has bugs but no CLIENT users exist yet | Medium | None (now), High (Phase 4) | Write unit tests for `getTasksForClient` using test CLIENT user even before Phase 4 ships. |
| Comment.internal field default (false) causes issues with existing comments | None | None | Default is false, which means "visible to everyone" — exactly how all existing comments already behave. |

### PHASE 3: Invitation System

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Auth.ts `createUser` event change breaks normal signup | Medium | High — new users get no workspace | Guard the change: only skip workspace creation if a matching Invitation record exists for that email. If no invitation → existing behavior (create workspace, assign OWNER). |
| Invite token brute force | Low | Medium — unauthorized access | CUID tokens are 25 characters, ~150 bits of entropy. Rate limit the `/invite/[token]` page. Add expiry check. |
| User already exists with that email in another workspace | Low | Medium — error on accept | Check for existing user at accept time. If exists and already has a workspaceId, show error: "This email is already associated with another workspace." |

### PHASE 4: Client Portal

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| CLIENT redirect in `(app)/layout.tsx` breaks something for OWNER/TEAM | Low | High — OWNER/TEAM get redirected to portal | The check is `if (role === 'CLIENT') redirect('/portal')`. OWNER and TEAM have role OWNER/TEAM — they won't match. But: if session.user.role is undefined (DB issue, stale JWT), they won't be redirected. Safe. |
| Client sees data they shouldn't (query filter bug) | Medium | High — privacy breach | The `getTasksForClient` query goes through ProjectMember. If no ProjectMember record, no results. Fail-closed. Test: create CLIENT with no ProjectMember records — should see empty portal. |
| Client navigates directly to `/tasks/[id]` (bypassing portal) | Medium | Medium — sees OWNER layout | The `(app)/layout.tsx` redirects CLIENT to portal. But: if they hit the URL before the layout loads (deep link), the page server component runs. Add role check in `(app)/tasks/[id]/page.tsx`: if CLIENT, redirect to `/portal/tasks/[id]`. |

### PHASE 5: Owner Dashboard Upgrade

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Dashboard changes break existing home page for OWNER/TEAM | Low | Medium — broken home | The change is additive: new sections replace "Today's Focus" section. Quick actions and activity are preserved. If the new query functions throw, wrap in try/catch and fall back to existing Today's Focus. |
| "Waiting on Clients" section is empty (no CLIENT users yet in workspace) | Expected | None | Section is conditionally rendered — if 0 results, section is hidden. Documented in USER-FLOWS.md. |
